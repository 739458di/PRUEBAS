💥 4. PUNTOS DE CRASH — Dónde se rompe el sistema
🔴 CRASH 1: reqPath is not defined L.6855
Dónde: HTTP server handler — cuando un request HTTP llega que no matchea /status, /api/send, etc.
Causa: Variable reqPath nunca se declara. Debería ser req.url.
Impacto: Genera ReferenceError continuo en logs. No mata el proceso pero llena logs de basura.
Fix: Cambiar reqPath → req.url en L.6855
🟢 CRASH 2: msg is not defined (YA FIXED)
Dónde: processBuffered() — seller lead detection
Causa: var _rawMsgForLead = msg; — msg no existe en scope de processBuffered
Impacto: ❌ CRASHEABA TODO processBuffered → mensajes de nuevos contactos se perdían silenciosamente
Fix: ✅ Ya arreglado — ahora usa rawMsgs[0] del buffer
🟡 CRASH 3: Claude API timeout/failure L.5100
Dónde: onMessage() → fetch a api.anthropic.com
Causa: API lenta, timeout, rate limit, o error 500
Impacto: Circuit breaker se activa → TODOS los mensajes se skipean por 5 minutos
Protección: AbortSignal.timeout(15s), circuit breaker, _failedMessageQueue para retry
🟡 CRASH 4: Baileys desconexión L.6170
Dónde: connection.update event
Causa: WhatsApp cierra sesión, phone se desconecta, ban temporal
Impacto: Bot offline, no recibe mensajes
Protección: Watchdog + auto-reconnect cada 30s. PM2 restart en crash fatal.
🟡 CRASH 5: DB (Turso) offline Varias
Dónde: Cualquier llamada a db()
Causa: Turso timeout, token expirado, red
Impacto: Mensajes no se guardan → recovery no los encuentra
Protección: try/catch en cada db call. Mensajes siguen procesándose aunque DB falle.
🟡 CRASH 6: processBuffered uncaught error
Dónde: Cualquier error no capturado dentro de processBuffered
Causa: Cualquier throw/ReferenceError/TypeError no envuelto en try/catch
Impacto: El setTimeout callback muere → mensaje se pierde. process.on('unhandledRejection') lo loguea pero no lo salva.
Protección parcial: La failsafe en messages.upsert guarda el mensaje raw en DB antes del buffer.

🛡️ 8. Triple Failsafe — Protección contra pérdida de mensajes
Failsafe 1: Catch en messages.upsert
El for loop tiene try/catch individual por mensaje. Si un mensaje crashea, los demás siguen procesándose. El catch guarda el mensaje fallido en wa_messages para recovery.

catch(_msgErr) { /* save to DB for recovery */ }
Failsafe 2: Catch global del batch
Todo el messages.upsert handler tiene un try/catch externo. Si el for loop entero falla, se loguea el error completo.

try { for (const msg...) {} } catch(outerErr) { log }
Failsafe 3: Recovery periódico
Cada 5 minutos, recoverMissedMessages() busca mensajes IN sin respuesta OUT en las últimas 2 horas. Re-procesa los que se perdieron.

setInterval(recoverMissedMessages, 300000)

🔧 10. SOLUCIÓN: Anti-Crash Architecture
Propuesta de Hardening
1. Wrap processBuffered en try/catch global
El problema más grande: Si processBuffered crashea, el mensaje se pierde silenciosamente porque el setTimeout callback muere sin recuperación.

Fix: Envolver TODO processBuffered en try/catch. En el catch: guardar los mensajes del buffer en wa_messages + programar recovery inmediato. 
async function processBuffered(key) {
  try {
    // ... todo el código actual ...
  } catch(fatalErr) {
    console.log('[FATAL-PB]', fatalErr);
    // Guardar mensajes perdidos para recovery
    var buf = msgBuffer[key];
    if (buf) {
      for (var m of buf.mensajes) {
        db('INSERT INTO wa_messages...',
          [buf.realPhone, m, 'in', ...]);
      }
    }
    delete msgBuffer[key];
  }
}
2. Fix reqPath → req.url (L.6855)
Impacto: Genera ReferenceError constante en logs. Fácil de arreglar.

// L.6855: cambiar reqPath → req.url
3. Heartbeat DB save cada 30s
Guardar estado del buffer en DB periódicamente. Si PM2 reinicia, recovery tiene datos frescos en vez de depender solo de wa_messages.
4. Validar variables antes de usar
Antes de acceder a cualquier variable en processBuffered, verificar existencia: 
// En vez de:
var x = msg; // crash si no existe

// Usar:
var x = (typeof msg !== 'undefined') ? msg : null;
